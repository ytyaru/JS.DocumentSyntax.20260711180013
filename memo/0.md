# DocumentSyntax

DocSynと略す。

文書構文。人間可読《ヒューマンリーダブル》なテキスト構文を定義するための体系を作る。

Markdownは人間可読な文書構文である。同様にWiki記法もそうだ。あのような人間がその原稿を読んで出力結果をイメージできる形式のテキストをここでは文書と呼ぶ。

文書には共通点がある。

1. 自然言語をベースに記述されること
2. 意味づけするためのメタ文字が混ざり込むこと

メタ文字は自然言語が邪魔されない程度であること。局所的な用途専門のドメイン言語を作るためのメタ言語とも言える。

## 全体像

No|責任区分|概要
--|---------|----
1|Normalizer|原稿を正規化する
2|Parser|原稿から構文木を返却する
3|Renderer|構文木から特定の構文形式テキスト(HTML等)を返却する
4|Pager|HTMLから画面サイズなどに応じて`<div>`要素でまとめた要素を返却する

今回のDocSynは1のNormalizerとParserのうち、文書構文のベースとなる基底構文と文法拡張を受け付ける機構を作り込む。

## テキスト形式

項目|値
----|--
文字セット|UTF-8
正規化形式|NFC
改行コード|LF
BOM|無し

DocSynの原稿は全てこの形式に従う。

## 正規化

Normalizerは渡された原稿テキストを正規化する。すなわち文字セット、Unicode正規化形式、改行コード、BOMを所定の形式に整える。

DocSynの本質は原稿を構文木に変換することだ。しかし原稿には構文エラーが含まれている可能性がある。その箇所を正確に示すためには文字位置を示す必要がある。だがバイナリデータの形式が違えば正常にテキストとして読み込めないし、改行コードが違えば正しく位置を算出できない。なので構文を定義する前に文字列データ形式を定める必要がある。その定義されたデータ形式と一致しているか判断し、一致させたデータに変換したデータを返却するのがNormalizerだ。

もし形式が不一致ならエラーで中断するなり自動修正したテキストを受け取ってダウンロードさせ、パースを再試行させること。

わざわざ再試行させるのは、パース処理において構文エラー時にエラー箇所を正確に示すためである。

例えばもしWindowsで改行コードCR-LFな原稿を投入され、それを自動的にLFに置換した場合、ユーザの手元にあった原稿と、内部で置換した原稿では文字数が変わってしまう。エラーが合った場合、その文字位置もズレてしまい、エラー箇所を特定できなくなってしまう。なのでテキスト形式はnormalizerで完全に一致させてからparserにかける必要がある。

## 文書構造

人間可読な文書の構造は、次の4構造に大別される。

* Fence
* Block
* Blank
* Inline

構造|概要
----|----
Fence|自然言語でない文法を許容する領域。複数行に跨る。自然言語のための構造であるBlock,Blank,Inlineとは違いメタデータを埋め込んだりCSVなどの構造化データやBase64画像を埋め込んだりソースコードのような異なる言語を埋め込んだりテーブルやリストの表現をさせたりする。Markdownと同じでありネストもできる。
Block|自然言語の段落や見出し、主題区切りなど、テキストにおける大きな塊を示す。二連続改行がメタ文字でありブロックの切れ目。
Blank|ブロック間改行。三連続以上の改行があれば一つ以上の空白となる。Web小説における大量の余白を改行で表現するためのもの。
Inline|ブロック内表現。連続した改行は存在しない。一つの改行はbr要素。`*強調*`などの改行を含まないもの。

```docsyn
---
フェンス
---

ブロックは
非連続な改行一つだけを
包含できる領域。

二連続以上改行でブロック分断する。


三連続以上改行があれば
改行数から二を引いた数だけ
ブロック間改行する。

改行はbr要素を入れることで実現する。これはブロック内改行、ブロック間改行に共通だ。

インライン要素は｜漢字《かんじ》や；傍点；や〔割注〕のように開始から終了までの間に一つも改行を含まない要素である。

----親
---子
子の内容
---

フェンスは行頭から特定の記号が3つ以上連続した時に開始される。これをフェンス開始記号と呼ぶ。これと同じものがそれ以降の行に最初に登場した時がフェンスの終了だ。それをフェンス終了記号と呼ぶ。

フェンス開始記号から改行するまでの間には、フェンス引数という任意のテキストを記入できる。これにより同じ記号であってもフェンスのパース方法を切り替えるなどの用途で使える。

フェンス開始記号や引数の終わりは改行コードで示される。そこからフェンス終了記号までの間が、フェンス内容だ。ここには二連続以上改行コードなども含めてそのまま記入できる。もしフェンス開始記号をそのまま記入したい場合は、親元のフェンス開始記号の記号数を増やして長くすることで重複を防いで実現できる。

フェンスを定義・復元するためには以下のような要素が必要だ。文法定義やToken作成などにおいて適宜必要データを渡すこと。

* フェンス記号
	* 文字
	* 個数
* フェンス引数
* フェンス内容
* フェンスパース関数
----
```

## Parser

原稿を1字ずつ読み進める。Tokenizerで種別ごとにテキストの開始位置と終了位置を取得し、その部分テキストからLexerがASTのNodeに変換する。

* Parser
	* Tokenizer
		* FenceTokenizer
		* BlockTokenizer
		* InlineTokenizer
	* Lexer
		* Lexer: `{kind:'fence', type:'heading', tokenize:(manuscript, pos, context)=>context.createToken(), lex:(token, context)=>context.createNode()}`
		* FenceLexer
		* BlockLexer
		* BlankLexer
		* InlineLexer
	* PluginManager
		* add, delete, get, gets
	* Context
		* 
	* Node
		* Fence
		* Block
		* Blank
		* Inline
			* Br
			* TextNode
* GrammarPattern
	* language: ja, en
	* Grammar
		* Tokenizer
		* Lexer
	* 例
		* en
			* NestedInline: `{{ {} }}` 英語圏ではメタ文字と内容を半角スペースで区切る
		* ja
			* NestedInline: `｛｛｛｝｝｝` 日本語圏では区切り文字はなく内容字数がゼロにならず閉じタグと数が合う最長をメタ文字とする

GrammarPatternは構文を定義するための糖衣構文である。同一アルゴリズムをGrammarPatternとして定義しておきメタ文字だけを任意に変更できるような構造にしておく。

## API

```javascript
const docSyn = new DocSyn(optionsD);
docSyn.plugin.gets({kind:'block'},{...}); // 配列（引数がなければ全件返却）。ゼロ時空配列。
docSyn.plugin.get({kind:'block', type:'heading'}); // 単数。ゼロ時null。
docSyn.plugin.add({kind:'block', type:'heading', tokenize:()=>{}, lex:()=>{}},{...});
docSyn.plubin.delete({kind:'block', type:'heading'},{...});
docSyn.plubin.fence/block/blank/inline.gets/get/set/delete(); // kindごとに各メソッド用意。
const res = docSyn.normalize(manuscript, optionsN);
docSyn.parse(manuscript, optionsP);
```
```javascript
new DocSyn({
    fences: [
        {type:'MyFence', chars:'-', arg:'フェンス引数', tokenize:(manuscript, pos, context)=>context.createToken(), lex:(token, context)=>context.createNode()},
        new grammarPatternA({...}),
    ],
    blocks: [
        {type:'MyBlock', tokenize:(manuscript, pos, context)=>context.createToken(), lex:(token, context)=>context.createNode()},
        new grammarPatternB({...}),
    ],
    blanks: [
        {type:'MyBlank', tokenize:(manuscript, pos, context)=>context.createToken(), lex:(token, context)=>context.createNode()},
        new grammarPatternC({...}),
    ],
    inlines: [
        {type:'MyInline', tokenize:(manuscript, pos, context)=>context.createToken(), lex:(token, context)=>context.createNode()},
        new grammarPatternD({...}),
    ],
});
```
